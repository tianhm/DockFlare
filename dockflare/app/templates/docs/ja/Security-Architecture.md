# DockFlare のセキュリティアーキテクチャと強化

このドキュメントでは、DockFlare 3.0+ において、Master ノードと登録済みエージェントの両方をどのように保護しているかを説明します。DockFlare に組み込まれている保護機構と、推奨される運用プラクティスを整理することで、セキュリティ監査を補完します。

## 1. コントロールプレーンの信頼モデル

- **Master を信頼の基点とする構成** – DockFlare Master は、Cloudflare の認証情報とポリシー定義をすべて保持します。Agent が API トークンを直接管理することはなく、認証済みチャネルを通じて受け取った指示だけを実行します。
- **Agent ごとの API キー** – 登録には Master が発行する一意の API キーが必要です。これらのキーは `agent_keys.dat` に暗号化された状態で保存され、所有者、タイムスタンプ、状態などのメタデータも一緒に管理されるため、いつでもローテーションや失効が可能です。
- **Master API の保護** – 管理画面や `/api/v2/*` を含む管理エンドポイントには、有効なセッションまたは Master API キーが必要です。トークンはレスポンスやログではマスクされ、スタックを再起動しなくてもローテーションできます。

## 2. 暗号化された設定と鍵管理

- **暗号化された `dockflare_config.dat`** – Cloudflare の認証情報、管理画面アカウント、トンネルのデフォルト設定、および Master キーは、`dockflare.key` で保護された暗号化 BLOB に保存されます。
- **暗号化された Agent レジストリ** – Agent API キーとその監査メタデータは `agent_keys.dat` に保存され、同じ Fernet キーで暗号化されます。機密情報は `state.json` には記録されません。
- **復元時の自動再起動** – バックアップアーカイブを復元すると、DockFlare は暗号化アーティファクトを書き込み、ランタイム状態を再読み込みし、再起動フラグを残して終了します。その後 Docker の再起動ポリシーにより、コンテナは新しい設定でただちに立ち上がります。
- **可観測性のための平文 `state.json`** – `state.json` は、オペレーターがルールや Agent の状態を確認しやすいよう平文のまま維持されます。一方で、シークレットに関しては暗号化ファイルが正本です。

## 3. バックアップと復元の保証

- **アーカイブの内容** – 各バックアップアーカイブ（`dockflare_backup_*.zip`）には `dockflare_config.dat`、`dockflare.key`、`agent_keys.dat`、`state.json`、およびチェックサムとバージョン情報を含む `manifest.json` が含まれます。Master ノードの再構築に追加ファイルは不要です。
- **自動化された復元フロー** – セットアップウィザードまたは Settings ページから復元すると、必要なアーティファクトを書き込み、ランタイムキャッシュを再読み込みし、コンテナを強制再起動して暗号化設定を直ちに適用します。
- **レガシー互換性** – 単体の `state.json` アップロードは、トラブルシューティングや部分移行のために引き続きサポートされています。DockFlare はランタイム状態のみを取り込み、既存の暗号化設定は保持するため、認証情報が誤ってリセットされることはありません。

## 4. ネットワークと通信のセキュリティ

- **Cloudflare Tunnel による通信経路** – Agent は受信ポートを公開しません。すべての通信は Master が管理する Cloudflare Tunnel を通るため、リモートホストの攻撃面を減らせます。
- **認証済みの Agent 呼び出し** – Agent の REST 呼び出しには API キーが含まれ、記録済みの Agent ID に結び付けられています。トークン不一致や失効済みキーは拒否されます。
- **Redis バックプレーン** – DockFlare は、キャッシュ、ログストリーミング、スレッド間シグナリングのために Redis を利用します。推奨 Compose スタックでは、Redis を専用の `dockflare-internal` ネットワークに配置し、`cloudflare-net` 上のワークロードから直接到達できないようにします。外部 Redis を使う場合は、認証と TLS で保護してください。
- **最小権限ランタイム** – Master と Agent はどちらも `dockflare` ユーザー（UID/GID 65532）で実行され、Docker との通信は付属の socket proxy 経由に限定されます。これにより公開される API の表面積を最小限に抑えられます。

## 5. 認証と認可

- **強化された管理画面ログイン** – 初期設定ウィザードでは、管理画面の管理者アカウント作成が必須です。パスワードログインを無効にすることもできますが、**Docker ネットワーク上の安全性に重大な影響があるため強く非推奨**です。
- **セッション管理** – Flask-Login セッションは暗号化された設定と結び付いています。バックアップの復元や認証情報のローテーションを行うと、既存のセッションは自動的に無効になります。
- **Agent ACL** – 各エージェントのレコードは、トンネル割り当て、ハートビートの時刻、保留中コマンドを追跡します。Master は、正しいトークンと登録済みステータスを持つエージェントにのみコマンドを配信します。

### ⚠️ 重要: 「Disable Password Login」に関するセキュリティ警告

DockFlare には、Cloudflare Access のような外部認証レイヤーで保護された高度な構成向けに **Disable Password Login** という設定があります。**ほとんどの環境ではこの機能の使用を強く推奨しません**。

**有効にした場合のセキュリティリスク:**
- **すべての API エンドポイントが認証なしでアクセス可能になります**
- **Docker ネットワーク上での露出** – DockFlare がインターネット側で Cloudflare Access に守られていても、同じ Docker ネットワーク上のコンテナはその外部認証を迂回して API に直接アクセスできます
- **アプリケーション側で認証を強制しない** – アプリケーションは、外部認証レイヤーがすべての安全性を担っている前提で動作します

**攻撃ベクトルの例:**
```
Internet → Cloudflare Access (Protected) → DockFlare ✅
         ↓
Docker Network → Other Container → DockFlare API (Unprotected) ❌
```

**推奨されるアプローチ:**
パスワード認証を無効にする代わりに、次のいずれかの安全な方法を使ってください。
1. **ローカル DockFlare 認証情報** – DockFlare に組み込まれたシンプルなパスワード認証
2. **OAuth/OIDC プロバイダー** – Google、GitHub、Azure AD などを使い、セキュリティを損なわずにシングルサインオンを実現する

どちらの方法でも適切な認証を維持しつつ、SSO の利便性を活かせます。OAuth を使えば、認証無効化に伴うリスクなしにシングルサインオンを導入できます。

**要点:** ネットワーク分離を含む非常に明確で十分に理解されたセキュリティアーキテクチャがない限り、パスワードログインは有効のままにし、利便性のために OAuth を利用してください。

## 6. 監査と運用上の可視性

- **メタデータの追跡** – Agent キーには `created_at`、`last_used_at`、`bound_agent_id`、状態、失効イベントが記録されます。`state.json` にも Agent の最終確認時刻が反映されるため、健全性を一目で確認できます。
- **ログストリーミング** – リアルタイムのログは Redis pub/sub を通じて配信されます。トークンやキーなどの機密値はクライアントへ届く前にマスクされます。
- **ステータス API** – `/api/v2/overview` は、トンネル、Agent、設定の状態をまとめて返し、監視システムや GitOps ワークフローで利用できます。

## 7. デプロイに関する推奨事項

| 項目 | 推奨事項 |
| --- | --- |
| Docker ボリューム | 暗号化設定、キー、状態を保持するために `/app/data` を永続化してください。ファイルログを有効にしている場合は `/app/logs` も永続化し、ホストマウントが UID/GID 65532 または上書きしたビルド引数で書き込み可能であることを確認してください。 |
| Redis | `redis:7-alpine` を DockFlare と同じプライベートネットワーク（`dockflare-internal`）で動かすか、認証/TLS で保護された Redis インスタンスを `REDIS_URL` に指定してください。Redis を公開ネットワークにさらさないでください。`REDIS_DB_INDEX` を使うと、同一 Redis を共有する他コンテナと DockFlare のデータを分離できます。 |
| バックアップ | `.zip` は定期的にダウンロードし、`dockflare.key` と一緒に保管してください。復元時に設定を復号化するには両方のファイルが必要です。 |
| Agent | API キーは機密情報として扱ってください。必要な Docker エンドポイントだけが公開されるよう socket proxy と一緒にデプロイし、コンテナが非特権ユーザー `dockflare`（UID/GID 65532）で動作することを前提に、ホスト権限を調整するか `DOCKFLARE_UID/DOCKFLARE_GID` を合わせて再ビルドしてください。 |
| リバースプロキシ | DockFlare は Cloudflare Access または別の信頼できる IdP の背後に配置してください。パスワードログインを無効にする場合は、上流の認証が常に確実に適用されるようにしてください。 |
| 監視 | 想定外の再起動、エージェントのハートビート欠落、メンテナンス時間外の新規キー発行に対してアラートを設定してください。 |

## 8. 今後の強化（ロードマップ）

- 保存時の Fernet キーに対するオプションのパスフレーズ保護
- 段階的ロールアウトを考慮した猶予期間付きの Agent キーローテーション自動化
- 読み取り専用操作と変更操作を分離するための、より細かな Agent コマンド権限
