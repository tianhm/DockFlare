# アクセスポリシーのベストプラクティスと例

DockFlare の中核となるセキュリティ機能が **Access Groups** です。Cloudflare Zero Trust を使ってサービスを保護するための、集中管理できて再利用しやすい仕組みです。

## 「黄金律」: Access Groups を使う

最も重要なベストプラクティスは、**共通のアクセス要件は必ず Access Groups にまとめる**ことです。

Access Groups は、DockFlare の管理画面で作成するポリシーテンプレートです。各コンテナに複数ラベルで複雑なルールを書く代わりに、ポリシーを一度作成し、`dockflare.access.group` の 1 行で適用できます。DockFlare v3.0.3 では、各グループを再利用可能な Cloudflare Access Policy に同期できるため、同じ判断ルールを複数アプリに再利用できます。

---

## アクセス グループを作成および使用する方法

Access Group は DockFlare の管理画面だけで完結して作成できます。

### ステップ 1: アクセス グループを作成する

1. DockFlare 管理画面のメインナビゲーションから **アクセス ポリシー** ページへ移動します。
2. **「アクセス グループの追加」** ボタンをクリックします。
3. グループに **一意でわかりやすい ID** を付けます。これは Docker ラベルで参照する ID です（例: `admin-users`、`home-network`、`geo-block`）。
4. モーダル上部のタブで **Access Mode** を選択します。
    * **認証済み（Authenticated）**: ユーザーはサインインが必要で、Cloudflare の `allow` 判定を発行します。
    * **パブリック（Public）**: `bypass` 判定を使い、アプリは公開のまま（必要に応じて Geo フィルターは適用）になります。
5. 選択したモードで表示される入力を埋めます（Authenticated はメール、どちらも国リストは任意）。
6. Authenticated の場合は、セッション期間、App Launcher 表示、IdP への自動リダイレクトなどの設定も調整します。
7. 保存します。DockFlare は定義をローカルに保存し、`DockFlare-AccessGroup-<id>` として Cloudflare に同期します。

### ステップ 2: アクセス グループを適用する

Access Group を作成したら、サービスへ適用する方法は 2 つあります。

#### A) Docker ラベルを使用する (推奨される方法)

新規/既存どちらのコンテナでも、作成した ID を `dockflare.access.group` に指定するだけです。

```yaml
services:
  grafana:
    image: grafana/grafana
    labels:
      - "dockflare.enable=true"
      - "dockflare.hostname=monitoring.example.com"
      - "dockflare.service=http://grafana:3000"
      # Apply the entire policy with one simple label:
      - "dockflare.access.group=admin-users"
```
`dockflare.access.groups` を使えば、コンマ区切りで複数のグループも適用できます。
`dockflare.access.groups=admin-users,home-network`

#### システム管理ポリシー

DockFlare には、常に利用できる 2 つの組み込みシステムポリシーがあります。

- **`public-default-bypass`** - `bypass` 判定によるパブリックアクセス（本当に公開したいサービス向け）
- **`authenticated-default`** - ワンタイム PIN + メール制限によるデフォルト認証

これらは削除できず、ゾーン保護やレガシーラベル移行の基盤として使われます。

#### B) 管理画面から設定する場合（手動ルールまたはオーバーライド）

ダッシュボードから任意のルールへ Access Group を直接適用することもできます。
1. メイン ダッシュボードで変更する入口ルールを見つけます。
2. **「ルールの管理」** をクリックします。
3. 編集モーダルで、[アクセス グループ] ドロップダウン メニューから目的のアクセス グループを選択します。
4. 保存します。

これは、手動ルール（Docker 外のサービス）にポリシーを適用したい場合や、Docker ラベル定義のポリシーを一時的に UI 側で上書きしたい場合に便利です。

## ポリシーの例

アクセス グループ内で作成できる一般的なポリシー構成をいくつか示します。

### 例 1: メールで認証する

これは最も一般的な使用例です。構成された ID プロバイダー (Google、GitHub、または電子メールに送信されるワンタイム PIN など) で認証できる特定のユーザーのみを許可します。

* **グループID:** `admin-users`
* **モード:** *認証済み*
* **許可される電子メール:** `user1@example.com`、`user2@example.com`
* **セッション期間:** `24h`

DockFlare は、指定したメールに対して `allow` 判定を発行し、それ以外はフォールスルーで `deny` となる再利用可能ポリシーを作成します。グループは `dockflare.access.group=admin-users` で適用します。

### 例 2: 自宅 IP を許可する

このポリシーはホーム ネットワークへのアクセスを制限し、他の場所で認証を強制しながら、信頼できる IP を使用している場合はログイン プロンプトをスキップできます。

1. **パブリック IP を見つける:** ブラウザで、「what is my ip」を検索します。パブリック IP アドレスが表示されます (例: `203.0.113.55`)。
2. **アクセス グループを作成します:**
    * **グループID:** `home-network`
    * **モード:** *認証済み*
    * **許可される電子メール:** `you@example.com`
    * **IP をバイパスする:** `203.0.113.55/32` を IP 許可リスト フィールドに追加します

DockFlare は、最初に IP 範囲をバイパスし、次にリストされた電子メールの認証を要求するポリシーを生成します。他の全員は拒否決定を受け取ります。

### 例 3: ジオフェンシング (複数の国をブロックする)

このポリシーは、特定の地域からのトラフィックを制限しながら、マーケティング サイトを公開したままにします。

* **グループID:** `public-eu`
* **モード:** *パブリック*
* **ブロックされている国:** `RU`、`CN`、`KP`

結果として得られる再利用可能なポリシーは、リストされている国を除くすべての人に対して Cloudflare `bypass` 決定を発行します。追加のコントロールを階層化する必要がある場合は、他のグループと組み合わせてください (`dockflare.access.groups=public-eu,admin-users`)。

## ゾーンのデフォルト ポリシー - セキュリティのベスト プラクティス

### ゾーンのデフォルト ポリシーとは何ですか?

ゾーンのデフォルト ポリシーは、まだ明示的に構成していないものを含む、DNS ゾーンのすべてのサブドメインを保護するワイルドカード `*.domain.com` アクセス アプリケーションです。

### なぜ必要なのか

**問題:** アクセス ポリシーをサービスに追加するのを忘れた場合、そのポリシーはデフォルトで公開されます。

**解決策:** ゾーンレベルのワイルドカード ポリシーはセーフティ ネットとして機能します。 `forgotten-service.yourdomain.com` の構成を忘れた場合でも、`*.yourdomain.com` ポリシーがそれをキャッチします。

### 設定方法

1. **アクセス ポリシー** ページに移動します
2. **ゾーンのデフォルト ポリシー (*.tld ワイルドカード)** セクションまでスクロールします。
3.「保護されていません」⚠️バッジが付いたゾーンを探します
4. [**ポリシーの作成**] をクリックします。
5. 適切なアクセス グループを選択します。
   - **パブリック ドメインの場合:** `public-default-bypass` を使用します。
   - **内部ドメインの場合:** 認証ポリシーを使用する
   - **混合使用の場合:** 最も制限的なポリシーを使用してください

### ベストプラクティス

- ✅ **運用ドメインのゾーン ポリシーを常に作成**
- ✅ **内部/プライベート ゾーンに認証ポリシーを使用**
- ✅ **真にパブリックなゾーンに対してのみパブリック バイパスを使用します**
- ✅ **定期的に確認** - ゾーンの保護ステータスを毎月確認します
- ⚠️ **優先順位を覚えておいてください** - 特定のホスト名ポリシーはワイルドカード ポリシーをオーバーライドします

### ポリシーの優先順位

Cloudflareはアクセスポリシーを次の順序で評価します。

1. **ホスト名が完全に一致** (例: `app.example.com`) - 最高の優先順位
2. **ワイルドカード一致** (例: `*.example.com`) - フォールバック
3. **一致なし** = パブリック アクセス (Access アプリなし) - デフォルト

これは、制限的なゾーンのデフォルト ポリシーを設定しながら、個々のサービスに対して特定の例外を作成できることを意味します。

## 外部 Cloudflare ポリシーの管理

### ポリシーの種類について

DockFlare では、[アクセス ポリシー] ページに 3 種類のポリシーが表示され、それぞれに視覚的なバッジが付いています。

- **🟦 DockFlare** - DockFlare によって作成および管理されるポリシー (プレフィックス: `DockFlare-`)
- **🟪 外部** - DockFlare の外部で作成されたポリシー (手動または他のツール)
- **🟧 システム** - 削除不可能なシステム ポリシー (`public-default-bypass`、`authenticated-default`)

### 外部ポリシーの同期

デフォルトでは、DockFlare は `DockFlare-` プレフィックスが付いたポリシーのみをインポートします。これにより、ポリシー リストがクリーンな状態に保たれ、コンテナ インフラストラクチャに重点が置かれます。

**すべての Cloudflare ポリシー**（手動作成分を含む）を同期するには:

1. 環境変数を設定します: `SYNC_ALL_CLOUDFLARE_POLICIES=true`
2. DockFlare を再起動します
3. [アクセス ポリシー] ページで **[Cloudflare から同期]** をクリックします。

外部ポリシーには、紫色の **「外部」** バッジが表示されます。

### 外部ポリシーをインポートする理由

**長所:**
- Cloudflare Access セットアップ全体を完全に可視化
- 既存のポリシーを再作成せずに再利用します。
- 1 つのインターフェースで集中管理
- 任意のポリシーを任意のサービス (DockFlare 管理かどうか) に適用します

**短所:**
- 多数の外部ポリシーがある場合、ポリシー リストが長くなります
- DockFlare 以外のサービスで使用されるポリシーを誤って変更するリスク

### ポリシーを整理する

**プロのヒント:** `DockFlare-` プレフィックスを使用するように Cloudflare の外部ポリシーの名前を変更します。

Cloudflare ダッシュボードで外部ポリシー名を変更して整理できます。

1. **Cloudflare Zero Trust** でポリシーを開きます
2. `DockFlare-` プレフィックスを使用するように名前を変更します (例: `DockFlare-LegacyVPN` または `DockFlare-ThirdPartyApp`)。
3. DockFlare で **「Cloudflare から同期」** をクリックします
4. ポリシーは **DockFlare 管理** ポリシー (青いバッジ) として表示されます。

これにより次のことが可能になります。
- ✅ DockFlare に表示されるポリシーを一貫した命名でグループ化できる
- ✅ 種別ごとにフィルター/並べ替えができる
- ✅ 「DockFlare が管理するもの」と「DockFlare に表示されるだけのもの」を区別できる

### フィルタリングポリシー

**フィルター** ドロップダウンを使用して、特定のポリシー タイプを表示します。

- **すべてのポリシー** - すべてを表示します (DockFlare、外部、システム)
- **DockFlare 管理** - 青色のバッジが付いたポリシーのみを表示します
- **外部** - 紫色のバッジが付いたポリシーのみを表示します
- **システム** - システム ポリシーのみを表示します

### 安全機能

**外部ポリシーの保護:**

外部ポリシーを削除/編集しようとすると、DockFlare は次の警告を表示します。

> ⚠️ 警告: これは、DockFlare によって作成されたものではない外部ポリシーです。
>
> このポリシーを変更すると、DockFlare の外部のサービスに影響を与える可能性があります。
>
> 本当にそうですか？

これにより、他のツールや手動構成で管理されているポリシーを誤って変更する事故を防げます。

### ベストプラクティス

1. **デフォルト設定 (推奨):**
   - `SYNC_ALL_CLOUDFLARE_POLICIES=false` を維持します (デフォルト)
   - DockFlare が管理するポリシーのみが表示されます
   - 明確で焦点を絞ったポリシー リスト

2. **高度なセットアップ (パワー ユーザー):**
   - `SYNC_ALL_CLOUDFLARE_POLICIES=true` を有効にする
   - すべてのポリシーを 1 か所で表示および管理
   - 外部ポリシーの名前を組織の `DockFlare-` プレフィックスに変更します

3. **ハイブリッドアプローチ:**
   - デフォルトで同期を無効にしておく
   - Cloudflare で重要な外部ポリシーの名前を手動で `DockFlare-*` に変更します
   - 次回の同期後に自動的に表示されます

4. **ポリシーの命名規則:**
   ```
   DockFlare-AccessGroup-<id>     # Auto-generated by access groups
   DockFlare-<custom-name>         # Your renamed external policies
   <anything-else>                 # Pure external (only visible if sync enabled)
   ```
