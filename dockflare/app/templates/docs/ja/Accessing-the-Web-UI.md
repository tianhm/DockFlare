# 管理画面へのアクセス

DockFlare コンテナが正常に起動したら、管理画面にアクセスして設定を管理し、トンネルの状態を確認し、入口ルールを手動で構成できます。

## デフォルトの URL

デフォルトでは、DockFlare の管理画面はポート `5000` で利用できます。ブラウザーで次の URL を開いてください。

```
http://<your-server-ip>:5000
```

`<your-server-ip>` を、DockFlare が実行されているサーバーの IP アドレスに置き換えます。

## 初回セットアップ

初めて管理画面にアクセスすると、**初期セットアップウィザード**が表示されます。このウィザードでは次の設定を行えます。

1. 既存の DockFlare バックアップ アーカイブ (`dockflare_backup_*.zip`) から復元します。このオプションを選択すると、システムは暗号化された構成、状態、およびエージェント キーをインポートし、コンテナを自動的に再起動してそれらを適用します。
2. 管理画面の管理者アカウントとパスワードを作成します。
3. Cloudflare アカウント ID、ゾーン ID (オプション)、および API トークンを入力します。
4. トンネル設定を確認し、オンボーディング手順を完了します。

## ログイン

初期設定後は、管理画面にアクセスするたびにログイン画面が表示されます。セットアップ時に作成したパスワードでログインしてください。

## パスワードログインの無効化

DockFlare には、DockFlare 自体が外部認証レイヤー (Cloudflare Access など) によって保護されている高度な展開を目的とした「パスワード ログインを無効にする」設定が含まれています。 **ほとんどの展開ではこの機能を使用しないことを強くお勧めします**。

### この設定が存在する理由

Cloudflare Access またはアプリケーションに到達する前に SSO を強制する別の認証プロキシの背後で DockFlare を実行する場合は、DockFlare の組み込みパスワード ログインを無効にして二重認証を回避できます。

### 有効にした場合のセキュリティリスク

- ⚠️ **この設定が有効になっている場合、すべての API エンドポイントは認証なしでアクセス可能になります**
- ⚠️ **Docker ネットワークへの露出:** DockFlare がパブリック インターネット上の Cloudflare Access の背後にある場合でも、同じ Docker ネットワーク上のコンテナは外部認証をバイパスし、DockFlare の API に直接アクセスできます。
- ⚠️ **認証の強制なし:** アプリケーションは外部認証がセキュリティを処理していると想定します。

### 攻撃ベクトルの例

```
Internet → Cloudflare Access (Protected) → DockFlare ✅
         ↓
Docker Network → Other Container → DockFlare API (Unprotected) ❌
```

DockFlare がインターネットからの Cloudflare Access によって保護されている場合でも、同じ Docker ネットワーク上で実行されているコンテナはその保護をバイパスし、認証なしで DockFlare の API エンドポイントに直接アクセスできます。

### 推奨されるアプローチ

パスワード認証を無効にする代わりに、次の安全なオプションのいずれかを使用します。

1. **ローカル DockFlare 資格情報** - DockFlare に組み込まれた簡単なパスワード認証
2. **OAuth/OIDC プロバイダー** - セキュリティを犠牲にすることなく簡単にシングル サインオンできるように Google、GitHub、Azure AD、またはその他の ID プロバイダーを構成します ([OAuth プロバイダーのセットアップ](OAuth-Provider-Setup.md) を参照)

どちらのオプションも、SSO の利便性を維持しながら、適切な認証を提供します。 OAuth オプションを使用すると、無効な認証によるセキュリティ リスクなしでシングル サインオン エクスペリエンスが得られます。

### 結論

ネットワーク分離を備えた非常に具体的でよく理解されたセキュリティアーキテクチャがない限り、パスワードログインは有効のままにし、利便性のために OAuth を利用してください。
